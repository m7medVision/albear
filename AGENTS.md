# AGENTS.md

## What this is

A local-only encrypted secrets manager. One Go daemon owns the vault; every
client talks to it over a Unix domain socket, and every client↔daemon channel
is separately end-to-end encrypted with the Noise Protocol Framework.

There is no server, no cloud, no telemetry, and no network listener. If a
change introduces an outbound request or a listening port, it is wrong —
except the update check (`internal/update`), which is opt-out and touches only
the GitHub releases API.

Four binaries / clients:

| Component | Path | Role |
| --- | --- | --- |
| `vaultd` | `cmd/vaultd` | Daemon. The only process that opens the database. |
| `vault` | `cmd/vault` | CLI. Administration and terminal workflows. |
| `vault-native` | `cmd/vault-native` | Chrome native-messaging relay. Blind: only ever sees ciphertext. |
| Chrome extension | `extension/` | MV3 popup + content scripts. Speaks Noise to `vaultd` *through* the relay. |
| Desktop app | `desktop/` | Electron. Speaks Noise to `vaultd` directly over the socket. |

The relay is the important design point: it forwards opaque Noise frames
between Chrome and the daemon. It cannot read or forge traffic. Never "help"
it by parsing or decrypting payloads there.

## Invariants

These are load-bearing. Breaking one is a security bug, not a style problem.

1. **Only `vaultd` opens the database.** No client links the sqlite packages.
2. **Plaintext never touches disk.** Encrypt before persisting, always.
3. **Domain packages depend on nothing.** No SQL, HTTP, Chrome, CLI, or
   filesystem in `internal/*/domain`.
4. **All SQL lives in two files** — `internal/infrastructure/sqlite/sql/commands.sql`
   (writes) and `.../sql/queries.sql` (reads). Single-statement queries only.
   Everything under `sqlite/gen/` is generated by sqlc; never hand-edit it.
5. **Sessions are memory-only**, epoch-bound, and die on lock or restart.
6. **Fail closed.** On any doubt — unknown client, bad origin, integrity
   failure — deny and lock. Never degrade to a permissive path.
7. **Nothing automatic ever deletes the vault.** Suspicious activity locks it.
   Destruction is explicit, interactive, and password-gated.
8. **Secrets never reach logs**, error strings, or telemetry of any kind.
9. **Fill/reveal requires explicit user action** and an exact origin match,
   verified daemon-side — never trusted from the client.

## Architecture

Domain-Driven Design with bounded contexts, layered per context.

```
cmd/            binaries; process setup only
  └─> internal/daemon/          socket server, request handlers, service wiring
        └─> internal/<context>/application/   use cases, orchestration
              └─> internal/<context>/domain/  entities, rules, invariants
        └─> internal/infrastructure/          sqlite, crypto, transport, system, ipc
        └─> internal/adapters/protocol/       wire envelope (request/response/codes)
```

Dependency direction is one-way: `cmd → daemon → application → domain`.
Domain never imports upward or sideways into infrastructure.

### Contexts

| Context | Path | Owns |
| --- | --- | --- |
| Vault Security | `internal/vault/` | Vault creation, lock state, unlock, key derivation/wrapping, master-password change, panic lock. The central context. |
| Secret Catalog | `internal/records/` | Records: create/edit/delete, search, domain matching, revisions. |
| Client Access | `internal/access/` | Client registration, pairing approval, capabilities, sessions, revocation, handshake authorization. |
| Security Events | `internal/security/` | Audit event recording. |
| Backup | `internal/backup/` | Encrypted backup container: create, verify, restore. |
| Shared | `internal/shared/domain` | Cross-context primitives: ids, clock, errors, secret wrapper. |

`internal/backup` and `internal/adapters` have no `domain/` — they are
orchestration and wire-format only, and that is deliberate.

### How dependencies actually work here

Read this before "cleaning up" the layering. Two rules that look
inconsistent but are intentional:

- **Cross-context dependencies are inverted.** A context that needs another
  declares a small interface *in its own package* and accepts it. Example:
  `records/application.KeySource` is declared in `records` and satisfied by
  `vault/application`. The Secret Catalog never derives or persists keys — it
  borrows the capability while the vault is unlocked. Follow this pattern for
  new cross-context needs.
- **Persistence is not inverted.** Services take a concrete
  `*sqlite.Store`, and `application` packages import
  `internal/infrastructure/{sqlite,crypto}` directly. There are no
  repository interfaces and no port/adapter indirection for the database.
  This is a deliberate trade, not an oversight — do not add a repository
  interface layer "for cleanliness" or to make things mockable. Tests use a
  real sqlite store.

So: this is DDD with a pure domain and a Go-idiomatic application layer, not
textbook Clean Architecture. The rule that matters is invariant 3 — the
domain stays pure. The application layer is allowed to know about
infrastructure.

### Composition root

`daemon.New` (`internal/daemon/server.go`) constructs every context service
and wires lock hooks:

```go
vault   := vaultapp.NewService(store, nil)
records := recordsapp.NewService(store, vault, nil)   // vault satisfies KeySource
access  := accessapp.NewService(store, vault, nil)
vault.OnLock(records.ClearIndex)                       // lock tears down the index
vault.OnLock(sessions.InvalidateAll)                   // ...and every session
```

`cmd/vaultd/main.go` does process-level setup only: open the DB, chmod 0600,
migrate, load the static key, listen, serve. Add new services in
`daemon.New`, not in `main`.

### CQRS

Writes and reads are generated into separate packages from separate SQL
files (see `sqlc.yaml`): `sqlite/gen/command` and `sqlite/gen/query`. Keep the
split — a statement that mutates belongs in `commands.sql`, a plain `SELECT`
in `queries.sql`. `internal/records/application` mirrors the split in
`commands.go` / `queries.go`.

### Transport

`internal/infrastructure/transport/noise` implements both handshake modes:

- `Noise_XX_25519_ChaChaPoly_SHA256` — pairing (unpaired clients).
- `Noise_XXpsk3_25519_ChaChaPoly_SHA256` — paired sessions; PSK derived from
  the client credential, daemon static key pinned.

A client opens with a hello envelope (`{"v":1,"mode":"cli"|"pairing"|"paired"}`)
as prologue, then exchanges length-prefixed encrypted frames carrying
`internal/adapters/protocol` JSON.

CLI mode is auto-authorized for the same user: the socket checks peer
credentials (`internal/infrastructure/ipc`). That check is why the daemon is
Linux-only — see Gotchas.

## Commands

```sh
make build                 # go build ./cmd/...
make extension             # pnpm install && pnpm build in extension/
make test                  # test-go + test-ext
make vet                   # go vet ./... + tsc --noEmit in extension/
make cover                 # go test with -cover
make fuzz                  # every fuzz target, 30s each
make sqlc                  # regenerate sqlite/gen from sql/*.sql
make vectors               # regenerate Noise interop vectors from Go
lefthook install           # install the git hooks (no make target for this)
cd desktop && npm run build && npx jest   # desktop has no make target either
```

Releases are built by GoReleaser (`.goreleaser.yaml`), which owns the version
stamp — there is no Makefile variable for it. To reproduce a release build
locally without tagging or publishing anything:

```sh
goreleaser check                             # validate .goreleaser.yaml
goreleaser release --snapshot --clean --skip=publish   # → dist/
```

A hand-rolled stamp, if you need one:

```sh
go build -ldflags "-X github.com/m7medVision/albear/internal/version.Version=v1.2.3" ./cmd/...
```

`go install` cannot pass ldflags, so `internal/version` also adopts an exact
release tag from `runtime/debug` build info. Pseudo-versions and `+dirty` are
rejected there on purpose: they mean a development build, which must keep
reporting `dev` so `update.Checker` never phones home.

Fuzz targets (`go test -fuzz=<Name> -fuzztime=30s -run='^$' <pkg>`):
`FuzzParseOrigin`, `FuzzDecodeMetadata`, `FuzzDecodeSecret`,
`FuzzParseContainer`, `FuzzReadNativeMessage`, `FuzzReadFrame`,
`FuzzServerHandshakeHello`.

## Conventions

- **Commits: one Conventional Commits subject line, nothing else.** The
  `commit-msg` hook (`tools/hooks/commit-msg.sh`) rejects message bodies and
  `Co-Authored-By:` trailers. Types: `feat fix docs style refactor perf test
  build ci chore revert`. Merge/Revert/fixup!/squash! subjects pass through.
- Commit in small, coherent chunks — not one omnibus commit.
- Go: stdlib-first. Adding a module dependency needs a real reason.
- Comments explain *why* a constraint exists, not what the line does.
- Tests sit beside the code (`*_test.go`, `tests/` in the extension).
- Releases are unified: one `vX.Y.Z` tag builds Go binaries (plus deb/rpm and
  checksums), the extension zip, and the Linux desktop installer into a single
  GitHub release. GoReleaser creates it as a draft; the other jobs attach to
  it; a final job un-drafts. Tag an `-rc` first to exercise the pipeline —
  `prerelease: auto` keeps it out of `/releases/latest`, which drives the CLI
  update notice.

## Gotchas

- **Albear ships Linux-only, and that is a product decision, not just a build
  tag.** `internal/infrastructure/ipc` and `internal/daemon` are
  `//go:build linux` (peer-credential checks), so `GOOS=darwin|windows go
  build ./cmd/vaultd` fails by design. `vault` and `vault-native` still
  *compile* on all three — but nothing they need exists there: `system.Paths`
  is XDG-only, `system.CheckPrivate` assumes unix permission bits,
  `install/chrome.go` refuses non-Linux outright, and there is no daemon to
  dial. Releases therefore ship linux/amd64 and linux/arm64 only; shipping a
  darwin/windows archive would mean shipping clients that cannot work. Don't
  "fix" a cross-platform daemon build without a real peer-credential
  implementation *and* per-OS paths for that OS.
- **Two TypeScript Noise implementations exist** —
  `extension/src/noise/noise.ts` and `desktop/src/main/noise.ts` — and their
  `testdata/vectors.json` are byte-identical copies. Change one, change both.
  `make vectors` regenerates from Go and copies to both; CI fails if either
  checked-in copy drifts. Go is the source of truth for the wire format.
- **`sqlite/gen/` is generated.** Edit `sql/*.sql`, then `make sqlc`.
- **The extension uses pnpm; the desktop app uses npm.** Not interchangeable.
- **Extension CSP forbids remote code.** No CDN, no eval, no remote fonts —
  bundle everything.
- Desktop renderer never touches the socket or Noise state: those live in the
  main process behind the `window.albear` contextBridge API.
- `vault install <browser>` writes native-host manifests outside the repo
  (into browser config dirs). Prefer `--print-only` when testing.
- The daemon holds the vault key in memory only; a restart locks it. Tests
  that expect an unlocked vault must unlock it themselves.
